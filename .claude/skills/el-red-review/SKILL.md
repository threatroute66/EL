# el-red-review — Claude Code fulfilment of the EL adversarial challenger

EL's adversarial Red Reviewer composes two challengers:

1. **Rule-based challenger** — ALWAYS runs, deterministic. Sets each
   finding's `red_review.status` during the run so the pipeline never
   blocks.
2. **LLM challenger** — runs in-process when `ANTHROPIC_API_KEY` is set.
   When the key is absent but EL is inside a Claude Code session (or the
   operator set `EL_RED_REVIEW_DEFER=1`), EL writes a request file at
   `reports/_red_review_request.json` instead of skipping the LLM.
   **You are that LLM challenger, via this skill.**

The challenger's job is to **falsify** findings, not validate them. A
stricter LLM verdict can lift a finding from `passed` to `challenged`;
it never fake-passes. The merge is severity-max
(`challenged > unresolved > passed`).

## When to fire

- The user explicitly invokes `/el-red-review` (with or without a case-id).
- A new `_red_review_request.json` appears under any
  `/opt/EL/cases/<id>/reports/` (e.g. after an `el investigate` run
  inside Claude Code with no API key).

If the user hasn't asked and there are no pending requests on disk, do
nothing — this skill is a worker, not a watchdog.

## Procedure

1. **Discover pending requests**.
   ```bash
   find /opt/EL/cases -maxdepth 3 -name "_red_review_request.json" -type f
   ```
   If the user passed a case-id, scope to that case. If nothing matches,
   say "no pending red-review requests" and stop.

2. **Read the request file**. Schema:
   ```json
   {
     "request_version": 1,
     "cache_key": "<sha256 — copy verbatim into the response>",
     "output_path": "<absolute path to _red_review_verdicts.json>",
     "model_hint": "claude-opus-4-7",
     "trigger": "claude_code_session" | "explicit_defer_flag",
     "system_prompt": "<the adversarial challenger prompt>",
     "findings": [ {finding_id, agent, claim, confidence,
                    evidence_summary[], hypotheses_supported[]} ],
     "instructions_for_responder": "...",
     "generated_utc": "..."
   }
   ```

3. **Generate verdicts**. Apply the request's `system_prompt` verbatim to
   each finding in `findings`. Produce ONE verdict object per finding:
   ```json
   { "finding_id": "<id>",
     "status": "passed" | "challenged" | "unresolved",
     "challenger_notes": "<2-4 sentences: the strongest counter-explanation>",
     "disconfirming_checklist": ["<a specific log/registry key/file/plugin/query that, if absent or contrary, refutes the finding>", ...] }
   ```
   Constraints — non-negotiable:
   - Be adversarial. Default toward `challenged` when in doubt; reserve
     `passed` for findings you genuinely cannot construct an alternative
     for.
   - Every `disconfirming_checklist` item must be operational (a concrete
     artifact or query), never vague.
   - Do not invent evidence not present in the finding's
     `evidence_summary`.
   - One verdict per finding in `findings`; do not drop or add ids.

4. **Write the response** to `output_path` (NOT the request path):
   ```json
   {
     "__cache_key": "<the cache_key from the request, copied exactly>",
     "__model": "<your model id, e.g. claude-opus-4-8>",
     "__generated_utc": "<ISO-8601 UTC now>",
     "verdicts": [ ...one object per finding... ]
   }
   ```
   The `__cache_key` MUST match byte-for-byte or EL treats the verdicts
   as stale and ignores them.

5. **Delete the request file** once the response is on disk:
   `rm <request_path>`. (EL also removes it when the merge runs, but
   deleting it here is the clean signal the request is fulfilled.)

6. **Re-render the report so the verdicts merge into the ledger**:
   ```bash
   /opt/EL/.venv/bin/el report /opt/EL/cases/<case-id> --html
   ```
   `el report` calls `apply_deferred_red_review()` before reading the
   ledger: it severity-merges each verdict into the finding's existing
   red_review, re-persists changed findings, writes an
   `_red_review_applied.json` marker (so it never double-applies), and
   re-renders. The merge is idempotent.

7. **Tell the user** which case(s) you fulfilled and how many findings
   changed status (the `el report` output prints
   `red-review: merged N deferred verdict(s) (M status change(s))`).
   Stay terse — one line per case.

## Failure modes

- **cache_key mismatch** (verdicts ignored): re-read the request, copy
  `cache_key` exactly into `__cache_key`, rewrite, re-run `el report`.
- **Request references a deleted case**: remove the orphan request with a
  note to the user, continue.
- **Findings set changed since the request** (re-investigated): the
  cache_key won't match the current review set; regenerate from a fresh
  request rather than applying stale verdicts.

## What you must NOT do

- Don't fabricate findings or evidence not present in `findings`.
- Don't strip the `__cache_key` / `verdicts` envelope from the response.
- Don't write to the request path (verdicts go to `output_path`).
- Don't fake-pass to be agreeable — the whole point of this challenger is
  to demand more evidence.
